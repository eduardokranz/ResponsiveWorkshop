What is Responsive Web Design

 

It is an approach that aims to create websites that present its content adequately in a variety of devices, ranging from mobile devices to desktop computers.

But how did we get to Responsive Web Design?

 

Back in the day we used fluid layouts

  • Comes from table layouts
  • Really old fashion solution for a layout
  • Layout can break quite easily and limited control over the content
  • Flexible structure
  • Department of Finance

Fixed Layouts

  • Web 2.0 made it really popular
  • Lots of control over the content
  • Fixed structure
  • NSW Government

Adaptive Design

  • Iphones and Ipads started to become really popular
  • The website has defined screen sizes that it should adapt to
  • Flexible structure
  • Changes parts of the design to adapt to the device visiting the website
  • Cats Who Code Blog

Responsive Design

  • We can't define what devices will be visiting our websites
  • Mobiles, Tablets, Laptops, Desktops, Fridges, Cars, TV, Game Consoles
  • Flexible Structure
  • The website organically responds to a certains screen size or a change in size.
  • Trent Walton
  • Food Sense

Challenges in Responsive Web Design

 

1. Handling Images

  • Small screens, big screens, Retina and HiDPI screens.
  • Delivering big images to mobiles over 4G/3G
  • Scaling images in the screen

1. Handling Images

2. Device Limitations

  • Battery Life
  • Connectivity and Download Speeds
  • No Hover, no pointer, etc.

3. Preparing Content

  • Designing content that can be viewed in different devices & screens
  • Gather the content at an early stage of the project

4. Other Challenges

  • It is much harder to design and develop for a Response website
  • Create a "visual content delivery system"

It is hard but it has it's benefits

What is good about Responsive Web Design

  • Single code base. Update in one place, reach all devices
  • It is ready for future devices (to an extend)
  • More accessible
  • WOW factor of being able to resize your browser
  • Medium

Lunch Break

Exercise

  • Plan a website that should be available on Mobile devices, Tablets, Laptops and Desktops.
  • You have all the content for it
  • It could be a blog, portfolio, brochure website or other topic of your interest
  • You have got 20 min to list the steps that you would take to design it
  • You can also sketch what it would look like on the templates that you have on your table.

What did you come up with?

If your steps look like something like this:

  1. Sketch ideas
  2. Create design concepts
  3. Create final mockup
  4. Update final mockup after revision 20 times
  5. Give mockups to the developement team
  6. Pray they will deliver what you had in mind
  7. Listen to the client saying that it isn't what he expected
  8. Cry!

You are doing it the painful way.

These are some tools that you can use for your advantage

  • Sketching
  • Wireframes
  • Style Tiles
  • Prototypes and Mockups
  • Pattern Libraries

Sketching

  • Great for fast ideation and iteration
  • Low fidelity
  • Low cost
  • You can use templates to help out
  • Sketch Example 1
  • Sketch Example 2

Wireframes

  • More structured, possibility of using real content (text)
  • Low to Mid fidelity
  • Lowish cost
  • You can create them in 2 ways:
  • Code: Simplified version of the final website, with no styles and mainly using black, white and boxes
  • Image: Created using Balsamiq, OmniGraffle or other image editor
  • Wireframe Example

Style Tiles

  • Style Tiles are a design deliverable consisting of fonts, colors and interface elements that communicate the essence of a visual brand for the web.
  • Mid to High fidelity
  • Mid cost
  • You have a template
  • Main goal is to showcase the look and feel that you have in mind, not the final design
  • Great to do iterations as changes are focus on style not on a final design mockup.
  • Style Tiles

Prototypes and Mockups

  • High cost
  • High fidelity
  • High cost to make changes to mockups
  • Lower cost to make changes to a prototype
  • Mockups are static and precise
  • Prototypes are dynamic and can also be precise.
  • You can reuse the code from your prototype or use it as a stepping stone to try things before building the final website.
  • Reference Viewer

Pattern Library

  • It is a library of visual assets that will be used in the building process of the website. Preferably in HTML/CSS/JS, so developers can just copy and paste into their template.
  • High fidelity
  • Mid cost
  • Creates consistency in the implementation and the library becomes a reference for anyone involved in the project
  • Good codeing knowledge is needed
  • It is like creating a small Twitter bootstrap for elements of your website
  • Pure CSS

My Process

I do not have a defined process yet, as everyone one else in the industry I'm still experiementing with different tools.

  • Content Planning
  • Sketch my ideas
  • Wireframe (previously on Photoshop, but now I'm aiming to start coding it)
  • Style Tiles
  • Prototype or Pattern Library

Why I chose those steps?

  • Sketches: Quick to get ideas out of my head and onto paper
  • Coded Wireframes: Allow stakeholders to visualize the content in different devices
  • Style Tiles: Gives you a look and feel for the project, but allows the final design to come out at the end of the process
  • Prototype: Great to communicate behaviour and style
  • Pattern Library: Reusable code, defines the design for the project. This asset is more useful than a mockup as it can be easily updated and used to push changes to a series of website pages at once.
